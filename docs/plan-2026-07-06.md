## Workstream 1 — body-local name transfer for close-matched functions

**Problem:** 7,576 functions match closely but not exactly; today they
transfer only function name + params, then the LLM re-names the whole
body. Their body locals are the largest remaining rename-noise
population (most of ~20K noise hunks).

**Approach (precision-first, two layers):**

1. _Mechanical transfer where structure aligns._ For a close pair,
   align statements/AST subtrees between prior and new bodies; where a
   subtree matches structurally (same shape modulo identifiers),
   transfer the local bindings inside it through the same
   placeholder-slot bridging exact matches use, applied via
   `attemptValidatedRename` like every other rename. Anything
   unaligned is left for the LLM. A wrong transfer is worse than a
   missed one — alignment must require corroboration, not just
   position.
2. _Prompt adherence for the rest._ The LLM already receives the prior
   function's code (`priorVersionContext`) but doesn't reliably reuse
   its names. Tighten the batch prompt: explicit "reuse these names for
   unchanged logic" instruction with the prior's name list, and pass
   transferred names as fixed (already supported via
   `alreadyRenamed`).

**Tests/measurement:**

- Red/green unit tests on hand-built close pairs (statement inserted,
  statement removed, condition changed) asserting which locals transfer
  and which don't.
- Perturbation-lab fixtures for ground-truth precision measurement.
- A/B on the real pair with the matching-only harness where possible;
  full run only to confirm.
- Add the hermetic diff-noise regression test (fixture pair, mock LLM,
  assert only genuinely-changed functions produce diff hunks) — this is
  the guardrail for all three workstreams even without a renderer.

## Workstream 2 — LLM-call efficiency strategy

**Problem:** the collision-retry tail now dominates incremental runs
(~2h of B″'s 3h; ~950 functions; ~4M input tokens; 2–4 in flight), and
more than 90% of completion tokens across ALL requests are the gpt-oss
reasoning channel (measured; see docs/llm-server-tuning-brief.md).

**Levers, in order:**

1. _Retry batching:_ collect collision retries across functions into
   shared batches instead of per-function serial loops; keep the lanes
   full during the tail.
2. _Retry context diet:_ retries re-send the full used-names list and
   large prompts for 2–3 identifiers. Send only the conflict-relevant
   names (the colliding name + proximate scope names) and the minimal
   code snippet.
3. _Cheaper conflict resolution:_ after the first LLM retry fails,
   resolve algorithmically (`resolveConflict` suffixing) instead of
   more LLM round trips — measure how often retry #2+ actually
   produces a better name than suffixing.
4. _Reasoning effort low_ (client `reasoning_effort` or server
   default) — A/B name quality on the preact smoke first
   (baseline: 1m36s / 78 calls / 115.7K tokens).
5. _Determinism:_ temperature 0 for naming; makes reruns reproducible,
   which the diff goal ultimately needs.
6. Fold in whatever the ops agent finds server-side (prefix caching,
   structured-output backend, scheduler limits — see the tuning brief).

**Measurement:** wall clock + token totals on the preact smoke and on
an incremental real run; retry-phase share of wall clock in the
renderer output.

## Workstream 3 — matcher precision hardening

**Why this may be more important than it looks:** every wrong match is
a wrong name presented as _unchanged_ — a silent lie in the eventual
changeset, unfixable downstream. At 79% exact-transfer, even a 1%
false-positive rate is ~340 functions with confidently wrong names
per version bump, and errors compound version-over-version as wrong
names become the next run's prior. We currently have KNOWN mechanisms
for false matches, all with executed evidence in
docs/code-review-2026-07-04.md:

1. _Non-injective matching:_ two prior functions can claim the same new
   function; last writer wins, stats double-count. Enforce injectivity
   in `matchFunctions`, demote multi-claimed targets to ambiguous,
   delete the binding-path claim-count hedge.
2. _Cascade fallback widening:_ when a disambiguation stage empties the
   candidate set, the next stage runs on a WIDER set — a candidate
   rejected by strong evidence can win at a weaker stage. Treat
   empty-after-filter as a contradiction and stop.
3. _Singleton-bucket blind accept:_ a bucket with one old + one new
   function matches with zero corroboration. Post-hash-fix the buckets
   are more discriminating, but the accept logic is unchanged — gate on
   memberKey/propertyAccesses/externalCalls agreement, reject on
   contradiction.
4. _Vote propagation keyed by name string:_ phantom identifier pairs
   from placeholder slots can rename UNRELATED module bindings that
   happen to share a minified name; a single vote wins. Emit votes only
   for resolved bindings, key by binding identity, require ≥2 votes.
5. _Prior-input contract:_ an empty/unparseable prior silently becomes
   a full-cost zero-transfer run; no same-program sanity check exists.
   Fail fast on parse; warn-or-abort below a match-rate floor.
6. _Phantom graph edges:_ deleting matched nodes from the graph leaves
   dangling dependency edges; every dependent becomes unsatisfiable and
   is released unordered by the deadlock force-break — in exactly the
   high-transfer runs we now have. Stop deleting; mark done and assert
   graph closure at scheduler entry.
7. _Runtime assertions:_ injectivity, graph closure,
   placeholder-alignment, post-rename binding spot-checks — cheap
   fail-fast instead of silent drift.

**Measurement:** the 196 cascade-residual functions from the anomaly
A/B are the immediate testbed (`/tmp/exp013-anomaly/leg-*-fixed.json`);
`measure-close-match-anomaly.ts` re-runs matching-only in ~4 min.
Precision claims verified against perturbation-lab ground truth.

### Behavioral contracts that will surprise you

- LLM provider errors are CONTAINED: a throwing `suggestAllNames`
  yields unrenamed outcomes via the straggler pass; `processor.failed`
  stays 0. Don't write tests expecting LLM throws to fail runs.
- Generated output goes through two gates before write: re-parse and
  semantic invariants (free-name set + total binding count vs a
  pre-rename baseline, `src/output-validation.ts`). Both report
  per-file and set a non-zero exit code; output is still written.
- ALL renames must go through `attemptValidatedRename`
  (`src/rename/validated-rename.ts`) — never `scope.rename` directly in
  transfer code. New rejection reasons need no stats plumbing
  (Record-keyed).
- Hash/placeholder APIs take NodePaths (binding resolution needs
  scope); hashes are rename-invariant; property names and free
  identifiers are hash CONTENT. Do not hash detached nodes.
- The renderer's in-flight gauge undercounts during the retry tail;
  progress can look stalled while retries proceed (check log mtime and
  token counters before assuming a hang).
